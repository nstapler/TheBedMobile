Coupling

Data Coupling

  • The AutoDriving System provides unstructured data (images) or location information to the BedroomMobileManger.
  • The BedroomMobile Manager provides unstructured data for UI to display.
  • The BedroomMobileDockManager provides unstructured data for Docking Manager UI to display.
  • The UI provides unstructured data to the BedroomMobile Manager in the form of user inputs.
  • The UI provides unstructured data to the BedroomMobileDockManager in the form of user inputs.

Stamp Coupling

  • The BedroomMobileUtilities provides structured data to the BedroomMobileManager to inform the state of its utilities.
  • The BedroomMobileManager provides structured data to the AutoDriving System.
  • The Central Server provides structured data to the BedroomMobileUI in the form list of available Docking Terminals.
  • The BedroomMobileDockManager provides structured data to the Central server to inform the server of its schedule.
  • The User Class provides structured data to the BedroomMobileUI to use in authentication and loading any presaved travel history or preferences.
  • Trip reads structured data from Notification Preferences and Travel Preferences

Content Coupling

  • The Central server modifies the BedroomMobileDockManager’s schedule based on input from the BedroomMobileUI.
  • The Trip modifies Notification Preferences and Travel Preferences according to input form the BedroomMobileManager.

Control Coupling

  • The BedroomMobileUtilities calls on the BedroomMobileDockUtilities to begin loading and unloading procedures.
  • BedroomMobileManager calls on Trip to start and track trips in progress.

Cohesion

Functional Cohesion

  • User Interface provides user interaction functionality.
  • BedroomMobileManager manages the state of bedroom-mobiles including its utilities and travel status.
  • Central Server summaries information from all BedroomMobileDocks to allow the BedroomMobileUI to locate available Docks with queries and inform BedroomMobileDocks with reservation requests.
  • BedroomMobileDockSchedule summarizes the schedule information for a single Docking Terminal.
  • BedroomMobileDock Utilities allows control over a Docking Terminals utility status and connections for docking and undocking.
  • Trip Manager holds all information necessary to track a User’s trip including destination and preferences.

Design Assessment

Most of our design would support iterative development, and a small part of our design which is user interaction functionality can support incremental development.

According to the requirements, we have bedroom-mobiles and docking terminals, and each of these two groups are managed by its own manager. The reason why these parts of design support iterative development is that none of them can be performed independently. Bedroom-mobiles needs docking terminals to undock and dock, and docking terminals needs bedroom-mobiles to fill their space in order for docking process to complete. After docking process has been established, then we can add additional services such as trip planning and rerouting for bedroom-mobile manager, and docking terminals registering and searching system for docking manager.

And for user interaction functionality, we can use incremental development to design our UI first using fake data which will be replaced with true data as docking system completes, and then give UI prototype to our customer to see if it fits what he wants. Then, we gradually change every part of UI until our customer or user are satisfied.

Useful Design Patterns

Builder

A builder pattern will be useful for our system in for constructing Trip objects. Which combine the name of the trip, trip origin and destination terminals, a list of stops to make, a depart time and an arrival time. All of this information will be put in by the user, drawn from the central server, or determined by the Auto-Driving system, then stored as a single built object until it is needed.

Facade

The Bedroom Mobile Manager could make use of a facade design pattern for all of its interactions with the Auto-Driving system. The Auto-Driving system is an already constructed application which is used in normal vehicles like personal automobiles for automated transportation, but the Bedroom-mobile uses the features of this system as well as additional features of its own to simplify the instructions which would otherwise need to be entered into the Auto-Driving system by the user into a more convenient format for the user.

Momento

A momento design pattern would be useful in this system for long term storage of planned trips in a less volatile form of memory. Trip objects can be converted into text files through the momento pattern, and then converted back as the time the trip is scheduled for approaches.

Interpreter

In order to use the Auto-Driving system, an interpreter may be necessary, depending on what language the Bedroom-Mobile Application is programmed in and the available features of the Auto-Driving system API.

Observer

The Bedroom-mobile application may use an observer design pattern for monitoring the time as well as active trips being executed by the Auto-Driving system so that it can update the user appropriately and travel on an appropriate schedule.

Composite

Depending on the complexity of the hardware involved in managing the utilities of Bedroom-mobiles and Docking Terminals, the Utilities object, may benefit from being designed as a composite object. So that all utility information for the respective Bedroom-mobile and Docking Terminals can be contained in one place.

  • View administered docking terminals

  • Search for rentable terminals

  • View Current Utility Status

  • Get status of utilities from docking terminal

Bedroom-mobile Dock Manager

  • Confirm registration of new users

Central Server

  • Find rental locations
    • Input: Dates, location

  • Calculate route to destination

Exceptions

User Input Exception

These exceptions occur when the user’s actions or input data are out of system boundaries. This type of exception could occur with any property of the bedroom-mobile that is physical or can be accessed by the user, such as the Bedroom-mobile Manager and Docking Manager applications or the interior of the bedroom mobile. Some examples of this type of exception might include a user requesting a registered docking terminal using a dock ID that does not exist or if the user forces open the door of the bedroom-mobile before the bedroom-mobile’s status is flagged as parked/not-in-motion. Exceptions of this type can be difficult to handle since users can act unpredictably so they must be handled with extra attention to user accessible features. This means robust internal user input handling around things like the checking of the user input the Docking Manager and the bedroom-mobile manager’s access to docking terminals, utilities, the route, and bedroom-mobile furniture.

Mechanical Exceptions

Some of the less common exceptions relate to the different digital sensors and components that enable the bedroom-mobile to provide an autonomous experience. There are sensors within the utilities, the bedroom-mobile vehicle, and extendable furniture that relate component status and updates to the bedroom-mobile manager. Each sensor has a chance of malfunctioning and providing the bedroom-mobile with inaccurate data that would contribute to unexpected trip outcomes or user inconvenience. Mechanical exceptions can be prevented through notifying the user of recommended bedroom-mobile inspections dates, checking component status at trip planning and again before the auto-driving system is engaged to ensure accuracy of route preconditions.

External Feature Exceptions

External features refers to the entities which are utilized but not fully controlled by the bedroom-mobile like the Central Docking Terminal Server and the Auto-Driving system. These systems rely on their own resources and software for support and error detection. We plan to handle these exceptions by having backup routines in place. For example, the bedroom mobile will store the most current route locally for when the GPS connection of the Auto-driving system is unavailable and allow direct connection to docking terminals within a wifi-range when connection to the central server is disrupted.
